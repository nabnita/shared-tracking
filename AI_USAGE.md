# AI Usage Log

This document outlines the AI tools utilized, key prompts, and concrete instances where the AI generated flawed code that required intervention.

## AI Tools Used
- **Agent**: Antigravity IDE
- **Core LLM**: Gemini 3.1 Pro

## Key Prompts Used
1. "Initialize a FastAPI backend and a React/Vite frontend with a modular structure."
2. "Write an anomaly detection rule engine in Python that parses a CSV containing dates, amounts, payers, and split percentages, and yields `RowDecision` objects without crashing on malformed data."
3. "Configure Railway deployment files for a multi-service Docker environment with PostgreSQL."

## AI Errors & Interventions

While the AI successfully scaffolded the vast majority of the application, there were several concrete cases where it hallucinated or provided flawed configurations that had to be manually diagnosed and corrected:

### 1. The Async Alembic Trap
**What the AI produced:**
When generating the database configuration, the AI set up an asynchronous database connection (`postgresql+asyncpg://`) for the FastAPI application, which is best practice. However, it also blindly fed this exact same async connection string into Alembic's `env.py` for database migrations.

**How it was caught:**
When running `docker-compose up` locally, the backend container immediately crashed with an obscure `AttributeError: 'AsyncEngine' object has no attribute 'execute'`.

**What was changed:**
Alembic migrations run synchronously and cannot natively use the `asyncpg` driver in the standard configuration block. The AI's `env.py` was rewritten, and a separate `DATABASE_URL` (using the sync `postgresql://` driver) was defined in `config.py` exclusively for Alembic, allowing migrations to execute successfully.

### 2. The Pydantic DB Flush Timeout (500 Error)
**What the AI produced:**
In the CSV ingestion pipeline, the AI wrote code to persist the `ImportReport` object using `await db.flush()` rather than `commit()`, and then returned that object to the FastAPI route to be serialized by a Pydantic schema (`ImportReportDetail`).

**How it was caught:**
During live deployment on Railway, the API returned a `500 Internal Server Error` when uploading a CSV. Railway Deploy Logs revealed a Pydantic `ValidationError`.

**What was changed:**
The ORM model for `ImportReport` used a `server_default=func.now()` for the `imported_at` timestamp. Because the AI used `flush()` instead of `commit()` or `refresh()`, the timestamp generated by PostgreSQL was not loaded back into the Python object. Pydantic saw a `None` value for a required `datetime` field and crashed. The pipeline was modified to explicitly generate the timestamp in Python (`datetime.now(timezone.utc)`) before database insertion.

### 3. Docker `$PORT` Variable Firewall
**What the AI produced:**
When deploying the React frontend via a multi-stage Nginx Dockerfile, the AI assumed that Vite would natively have access to Railway's injected environment variables (like `VITE_API_URL`) during the `npm run build` step.

**How it was caught:**
The website successfully loaded in production, but API requests failed due to CORS and `ERR_CONNECTION_REFUSED` errors attempting to hit `http://localhost:8000`. The browser DevTools revealed that the build had fallen back to the local development URL.

**What was changed:**
Docker places a strict firewall between host environment variables and the `RUN` commands in a build phase. The AI's `Dockerfile` was modified to explicitly punch a hole in this firewall by adding `ARG VITE_API_URL` and `ENV VITE_API_URL=$VITE_API_URL` directly above the `npm run build` command, allowing Vite to successfully bake the production URL into the static JavaScript files.
